当项目里有十几个 skills，同一句“帮我发布文章”可能同时触发写作、构建、Git 和发布规则。

## 问题机制

技能多了以后，难点不再是写新能力，而是决定谁先运行、谁能覆盖谁。语义相似只能说明请求像某个技能，不能说明这个技能现在应该执行。发布文章既像写作任务，也像部署任务，还可能涉及敏感信息和远端状态。

```bash
rg -n "description:" ~/.codex/skills/*/SKILL.md
```

如果多个技能都声称能处理“发布”，路由器必须看风险，而不只是看关键词。

关键词路由还会制造误触发。用户说“审一下这篇文章能不能发布”，写作 skill 可能想改正文，发布 skill 可能想推送远端，审计 skill 可能只想检查风险。三者都相关，但只有一个应该先运行。

## 最小方案

最小方案是让每次路由输出候选、理由和风险等级。低风险候选可以直接跑，高风险候选要确认。

```javascript
const route = {
  skill: "tech-geek-blog",
  reason: "用户要求改写技术博客",
  risk: "writes_files",
  next: "draft_only"
};
```

如果下一步变成 `push_remote`，就必须切到发布边界，不能继续沿用写作技能的权限。

组合任务最好拆成执行图。每个节点只拿自己需要的权限。写作节点改 Markdown，构建节点跑 `npm run build`，发布节点确认分支和远端。节点之间传递的是产物和状态，不是无限制上下文。

```javascript
const graph = ["rewrite_post", "build_site", "confirm_push"];
```

## 边界风险

路由规则会在组合任务里失效。比如“修正文并发布”包含两个阶段，写作阶段允许改 Markdown，发布阶段需要构建和 Git 检查。最小验证是打印执行图，而不是只打印最终技能名。

```bash
node scripts/route.js "修正文并发布" --explain
```

核心原则是把技能选择变成可解释调度。多个 skills 不是工具箱越大越好，而是边界越清楚越可靠。

路由也会失效在用户覆盖上。用户临时说“别跑构建”，这个覆盖只应该影响当前任务，不能污染下次发布。最小记录是把覆盖写进 run log，而不是写回全局配置。否则一次临时省事，会变成长期风险。

冲突时要有优先级。安全规则高于写作风格，项目规则高于通用建议，用户本轮授权高于默认禁止但只在本轮有效。如果没有优先级，两个技能都以为自己在遵守规则，系统却给不出可解释结果。

```javascript
const priority = ["safety", "project", "user_turn", "style"];
```

还有一个细节是状态信封。一个 skill 的输出交给下一个 skill 时，不能只传正文。至少要带上来源、改动文件、验证状态和剩余风险。否则发布 skill 看到一篇 Markdown，不知道它是否已经构建通过。

```javascript
const envelope = { artifact: "post.md", verified: false, risks: ["not_built"] };
```

路由解释也要给用户看得懂。不要只显示 `score=0.82`，而要说明“匹配博客写作，但涉及发布，所以需要构建确认”。可质疑的解释比神秘分数更有用。

用户能质疑，调度才不会变成新的黑箱。

这也是多技能系统的最低可用标准和长期维护的底线。
