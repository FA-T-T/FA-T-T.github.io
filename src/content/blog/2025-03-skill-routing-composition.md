家里工具箱里有扳手、螺丝刀、电钻。会不会修东西，不取决于工具数量，而取决于你能否在正确时机拿出正确工具。2025 年 skill 路由系统面对的是同一种问题。

当 skill 增长到几十个，系统复杂度来自交叉影响：同一个用户请求可能同时命中多个技能，且每个技能都带自己的上下文策略和输出格式。没有路由层，系统会在“重复调用、冲突指令、状态污染”里快速失稳。

可执行路由通常采用多因子评分，不只看意图分类。典型因子包括：语义匹配度、工具调用成本、历史成功率、合规风险等级、当前系统负载。路由器输出应是候选列表与置信度，而不是单一拍板，这样才能支持降级和 A/B 选择。

覆盖机制决定“临时偏好”如何进入执行链。用户说“只用英文资料”，可以覆盖 context 过滤器；用户说“结果越快越好”，可以覆盖 `top_k` 或超时参数。覆盖规则必须声明作用域（本轮/本会话/全局）和生存期（TTL），否则一个临时偏好会污染后续任务。

链式调用把多步骤任务拆成可管理流水线。比如“法规检索 → 风险判定 → 执行清单生成”，每步输出都写入结构化状态容器，例如 `state.envelope = {artifacts, citations, decision_log}`。下一步只读必要字段并追加新记录，这样回放时可以精确定位哪一步偏离。

冲突消解需要确定性规则。实践里常见“优先级 + 不可覆盖字段”：高优先级 skill 可覆盖低优先级参数，但不得覆盖系统安全约束、审计字段和法务红线。规则越明确，线上故障越容易复盘。

业内存在两种偏好：一类推崇“智能路由器全自动决策”，一类坚持“人工编排固定链路”。前者可取在适应性，弱点是难解释；后者可取在稳定性，弱点是扩展慢。成熟系统一般采用混合模式：高风险任务固定链路，低风险任务动态路由。

回到生活，这和医院分诊、物流分拨、客服分流是同一个工程母题：有限资源下的正确分配。skill 组合的价值不只在效率，更在可控与可追责。
