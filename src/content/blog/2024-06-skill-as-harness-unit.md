一个团队把提示词、脚本和参考资料分别放在三个地方，换人维护时工作流立刻失忆。

## 问题机制

技能的意义不是名字，而是把一类任务打包成可维护单元。只靠提示词会膨胀，只靠脚本又缺少任务语义。比如写博客需要风格规则、事实约束、文件模板和发布检查。如果这些散在聊天记录里，下次执行只能重新拼。

```bash
tree skills/tech-blog
```

如果目录里只有一段长提示词，没有触发条件、示例和失败边界，它就不是技能，只是备忘录。

散装工作流还有迁移问题。写作者知道要先读风格文件，工程师知道要跑检查脚本，新人却只看到一个“帮我写博客”的提示词。知识没有进入目录结构，就会留在人脑里。人一换，流程就断。

## 最小方案

最小技能可以只有一个 `SKILL.md`。它说明何时使用、输出什么、禁止什么。重复操作再放进脚本，不要让模型每次即兴重写。

```bash
mkdir -p skills/tiny-blog
cat > skills/tiny-blog/SKILL.md <<'EOF'
# Tiny Blog
输出 800-1200 字中文技术博客
必须含 frontmatter 和 social 摘要
EOF
```

这个结构很小，但已经能被复制、评审和版本化。

如果任务需要稳定检查，再补一个脚本，而不是把检查规则写成长段自然语言。比如博客 skill 可以要求 frontmatter，并用脚本检查标题长度。模型负责写，脚本负责判。

```bash
node scripts/check-post.js drafts/post.md
```

## 边界风险

技能太大也会失败。把所有风格、项目、工具和历史决策塞进一个文件，会让触发变慢，规则互相覆盖。截图检查很简单：打开 `SKILL.md`，如果三分钟内找不到输出格式和禁止事项，它就太臃肿。

```bash
wc -l skills/tiny-blog/SKILL.md
```

核心原则是把稳定任务知识放进文件，把确定性动作放进脚本，把临时目标留给 prompt。

技能的边界也不能太宽。一个同时会写博客、生成图片、部署站点、修 CI 的 skill 看起来强，实际会让权限混在一起。最小单元应该围绕一种产物和一组稳定规则。需要组合时，让路由层组合，不要让单个 skill 变成杂物间。

还有一种失败是把 skill 写成教程。教程给人看，skill 给 agent 用。它不需要解释太多背景，只需要告诉 agent 何时触发、怎么输出、哪些文件能读、哪些动作不能做。真正复杂的参考可以放在单独文件里，但默认不加载。

```bash
find skills/tiny-blog -maxdepth 2 -type f
```

技能还要有一个很小的 smoke test。比如给它一段短主题，要求产出 frontmatter 和三个小节。测试不是为了评价文采，而是确认 skill 没有忘记格式。每次改 skill 后跑一次，就能发现“输出格式被新说明冲掉”这种低级问题。

```bash
codex "用 tiny-blog 写一篇关于 git diff 的短文"
```

如果测试输出缺少 frontmatter，说明格式规则不够硬。如果输出跑到两千字，说明长度规则没有进入核心约束。小测试能暴露大漂移。

这些检查越小，越容易坚持。

坚持，技能才会变成资产。
