一次 agent 运行可能读文件、调工具、写结果，如果只检查最终回答，事故已经发生。

## 问题机制

安全不能只放在最外层。输入检查能拦住明显不该做的请求，但工具调用发生在中间，输出检查只能看到最后文本。比如模型读到了 `.env`，即使最终回答没有打印密钥，敏感信息也已经进入上下文。

```bash
rg -n "OPENAI|TOKEN|PASSWORD" . --glob '!node_modules'
```

这类命令只能检查变量名是否存在，不能把真实值打印给模型。

护栏还要分时机。输入护栏决定任务能不能开始，工具护栏决定动作能不能发生，输出护栏决定结果能不能交付。如果只在最后扫一遍文本，就拦不住已经发生的文件读取和远端写入。

## 最小方案

最小护栏是把读写权限、工具动作和交付报告写进技能规则。每次运行记录输入来源、工具名、改动文件和验证结果。

```python
audit = {
    "skill": "tech-geek-blog",
    "files_changed": ["src/content/blog/post.md"],
    "commands": ["npm run build"],
    "secrets_read": False,
}
```

这不是企业级审计系统，但足够让个人项目知道发生了什么。

更实用的做法是每次完成后列出事实，而不是写一句“已完成”。改了哪些文件，跑了哪些命令，哪些测试没跑，哪些风险还在。这些字段能让用户决定是否继续提交。

```bash
git diff --name-only
git status --short
```

## 边界风险

护栏会失效在两处。第一，规则太软，只写“注意安全”。第二，日志太重，把敏感内容完整记录下来。正确做法是记录路径、摘要和状态，不记录 secret 值。

```bash
git diff --name-only
npm run build
```

核心原则是把安全放到动作之前，把审计放到结果之后。能执行的技能，必须能停止，也必须能回看。

边界在于审计本身也会泄漏。不要把完整密钥、客户数据或私有文档内容写进日志。记录路径、哈希、状态和脱敏摘要就够了。护栏不是让系统什么都不做，而是让每一步都有可检查的停机点。

还要把发布动作单独看待。生成草稿、修改文件和推送远端不是同一种风险。一个博客 skill 可以写 Markdown，但不应该默认 `git push`。如果确实要发布，先跑构建，再展示改动范围，最后确认远端分支。

```bash
npm run build
git diff --stat
```

最小审计还要记录“没有做什么”。没有读 `.env`，没有安装依赖，没有推送远端，这些都应该写清楚。很多事故来自默认假设：用户以为 agent 没做，agent 以为没必要报告。把未执行动作列出来，可以减少这种灰区。

```python
audit["not_run"] = ["git push", "npm install", "cat .env"]
```

如果未来要自动化，先自动化低风险检查。比如检查标题、跑构建、列 diff。不要一开始就自动推送、自动部署、自动改权限。护栏的成熟顺序应该和风险相反。

低风险动作稳定后，再逐步放开更高权限。这样系统是在积累信任，而不是一次性赌运气。

没有这种顺序，护栏只是写在纸上的安慰，无法承担真实工作流程风险。
