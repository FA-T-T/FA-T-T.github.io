你把地址发给快递员，少一个门牌号，包裹就会在楼下绕圈。模型接工具也一样：能力不足常常不是“不会想”，而是“没拿到可执行参数”。2023 年函数调用协议的成熟，本质是把模糊文本交互改造成可验证 I/O。

早期插件生态验证了“模型可调工具”的方向，但也暴露了解析脆弱：字段漂移、类型错配、必填缺失。系统方很快从“解析自然语言 API 请求”转向“让模型填 Schema 对象”。JSON Schema 的 `required`、`enum`、`pattern`、`minimum/maximum` 把生成空间压进机器可校验边界。

这带来一个重要分工：模型负责意图到参数映射，执行层负责校验、调用、重试、回退。把责任分开后，系统才能解释“哪一步错了”。如果模型生成了 `date: "tomorrow"` 而 schema 要求 ISO 日期，执行层可以返回精确错误，再让模型定向修复，而不是盲目重生成整段。

并行工具调用是性能拐点。多源任务若串行执行，延迟近似各工具耗时之和；并行后接近最大单工具耗时。前提是工具之间没有隐式共享可变状态，且结果汇聚层能处理部分失败与超时。工程上常见做法是给每个调用附 `trace_id`、`timeout_ms`、`retry_policy`，汇聚后再统一交给模型归并。

输出端同样需要协议化。很多团队只规范输入，不规范输出，导致下游系统拿不到稳定字段。更稳妥方式是双向结构化：输入 schema + 输出 schema，并在输出里显式区分 `result`、`evidence`、`errors`。这使自动化测试和灰度发布都可度量。

业内对函数调用的分歧主要在“模型主导”还是“运行时主导”。前者可取之处是灵活，能适应新任务；不可取之处是行为漂移难审计。后者可取之处是稳定、可回放；不可取之处是初期配置成本高。成熟团队通常采用“模型做选择，运行时做边界控制”的混合架构。

回到生活，协议化思维适用于所有“人机协作 + 外部系统”场景：报销审批、客服工单、运维告警、合同审阅。你不是让模型“自由发挥”，而是让它在可验证轨道内高效执行。这就是 harness 能长期运行的基础。
