团队把同一段提示词复制到三个脚本里，模型升级后输出字段漂移，下游解析器先坏。

## 问题机制

提示词不是一句咒语，而是一次调用的接口。问题在于自然语言很容易把任务、变量、示例和输出格式混在一起。比如“总结会议并给出行动项”看似清楚，但没有说明行动项字段、证据来源和缺失信息时的处理方式。维护者只能靠猜。

```bash
rg "总结会议" prompts scripts
```

如果这条命令能搜到多个相似版本，就说明提示词已经像未封装的重复代码。一次改动不会自动同步，回归也无从判断。

更麻烦的是，这种坏法很隐蔽。模型仍然会给出流畅回答，页面上看不出接口已经漂移。真正报警的是下游脚本，比如原本读取 `owners`，新输出却写成“负责人”。这不是文案问题，而是字段契约断了。

## 最小方案

最小可落地做法是把提示词写成小型契约。它不需要复杂框架，只要把输入、输出和拒绝条件分开。

```python
prompt = f"""
任务: 把会议记录整理成决策备忘录
输入: {meeting_text}
输出: JSON, 字段为 decisions, owners, open_questions
规则: 没有证据的 owner 写 null, 不要猜测
"""
```

这样下游代码可以检查字段，而不是解析一段散文。提示词也能进 git diff，维护者能看见是改了输出字段还是改了语气。

如果团队里有多种任务，可以把契约放进文件，而不是散在代码字符串里。比如 `prompts/meeting-memo.md` 只管会议纪要，`prompts/bug-triage.md` 只管缺陷归类。脚本读取文件，再注入变量。这样 review 时看到的是模板变更，不是一段混在业务逻辑里的长字符串。

```bash
git diff prompts/meeting-memo.md
```

## 边界风险

提示词契约不能保证模型一定正确。它只让错误更容易定位。比如模型仍可能把“讨论过的想法”写进 decisions。此时要加验证，而不是继续堆角色设定。

```javascript
for (const item of result.decisions) {
  if (!meetingText.includes(item.evidence)) throw new Error("missing evidence");
}
```

核心原则是把提示词当作可维护接口。能命名输入，能检查输出，能说明失败，就比一段神奇措辞更像程序。

它仍然不是传统程序。模型不会因为你写了 JSON 约束就百分百遵守，也不会因为你写了“不要猜测”就完全停止推断。可维护性的意义是把错误变成可见对象。字段错了修 schema，证据缺了修验证，任务理解错了修示例。不要把所有问题都归因于模型笨，也不要把所有希望都压在一句更强的提示词上。

还要警惕过度模板化。把所有场景都塞进一个万能提示词，会让变量和规则互相踩踏。客服摘要、代码评审、实验记录应当有不同契约。复用的是结构，不是同一段话。

如果要做最小回归，可以保存三条真实输入和期望字段。每次改提示词后跑一遍，至少确认字段没有消失，拒绝条件仍然触发。它不证明答案完美，但能拦住最常见的接口漂移。
