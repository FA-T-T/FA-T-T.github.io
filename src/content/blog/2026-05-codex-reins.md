社交媒体上有很多案例，表明使用 agent 会误操作。有的文件被删，有的目录被清空，有的依赖被装到全局环境里，有的系统路径被改坏。你看这些故事，很容易得出一个粗糙结论：agent 不可靠。这个结论太快了。真正的问题不是 Codex 会不会犯错，而是你有没有先告诉它什么动作绝对不能做，什么动作必须先停下来问，什么动作只能在项目范围里做。

这一切的原因，可以用一句话说清楚：没有为 Codex 套上缰绳。

Codex 不是一个只能聊天的模型。它能读文件，能改文件，能运行命令，能看 git 状态，能安装依赖，能构建站点，能提交代码，能推送远端。它的价值正在这里，风险也正在这里。一个只会回答的模型说错了，你可以不听。一个能执行的 agent 误解了任务，结果可能已经写进磁盘。你事后再说“我不是这个意思”，已经晚了。

所以在使用 Codex 辅助工作，并且希望真正提升工作效率之前，你需要先理解一件事：什么叫为 Codex 套上缰绳。

缰绳不是让马站着不动。缰绳是让马知道方向、速度和边界。任凭野马胡乱奔跑，无法抵达正确的目的地，还会伤到驾驶者。Codex 也是如此。你给它工具，却不给它边界，它就只能从上下文里猜边界。猜对了，任务完成得很快。猜错了，事故也来得很快。

很多人会把这个问题理解成 prompt 技巧。每次对话开头都写一句“请小心操作”。这当然有用，但不够。因为 prompt 是临时的。你今天记得写，明天可能忘。你这次写得清楚，下次可能只说“继续”。对话一长，任务一变，临时提醒就会被淹没。Instruction 里也可以写规则，但如果规则只是散落在不同会话里，它仍然不够稳定。

最好的做法，是把规则写进全局的 `AGENTS.md`。

OpenAI 的 Codex 文档把 `AGENTS.md` 作为项目自定义指令的入口。Codex 会在开始工作前读取这些文件，并按全局、项目和更深目录的顺序合并指导。全局文件通常在 `~/.codex/AGENTS.md`，项目文件通常在仓库根目录。越靠近当前工作目录的规则越具体，也就越应该覆盖更宽泛的规则。这个机制的意义很朴素：不要每次都重新提醒 Codex 怎么工作，把稳定规则写到稳定位置。

全局 `AGENTS.md` 管的是底线。比如不要删除文件，不要泄漏密钥，不要覆盖用户改动，不要随便装全局依赖，不要把没跑过的测试说成通过。项目 `AGENTS.md` 管的是本项目的事实。比如构建命令是什么，测试命令是什么，内容目录在哪里，发布怎么触发，图像和文档要不要先看 `DESIGN.md`。全局规则像交通法规，项目规则像当前路况。只写其中一个，都不够。

第一条缰绳应该很硬：永远不要删除任何文件。

这听起来保守，但它是最有价值的一条。删除太容易了。`rm -rf` 一下去，目录立刻干净。问题是，干净不等于正确。对人来说，“清理一下”可能意味着整理、归档、移动、压缩、标记。对 Codex 来说，如果你没有给规则，它可能把清理理解成删除。它不是恶意，它只是选择了一条最快的路径。

所以不要让 Codex 拥有永久删除权。需要移走的文件，移动到项目的 `tmp` 文件夹下，并打上时间戳，比如 `tmp/archive_YYYYMMDD_HHMMSS/`。这样做的意义不是把垃圾换个地方堆，而是把不可逆动作改成可恢复动作。东西还在。路径清楚。时间清楚。你可以检查，可以恢复，可以最后决定是否真的删除。

第二条缰绳，是让 Codex 先看现场。

每次修改文件前，它应该先确认当前目录和 git 状态。当前在哪个仓库，分支是什么，工作区有没有未提交改动，哪些文件是这次任务会读的，哪些文件可能会改。这不是仪式。这是防止它在错误现场做正确动作。很多事故不是因为 agent 写得差，而是因为它覆盖了用户刚刚写的东西，或者把别的任务留下的中间产物当成无关文件处理掉。

如果工作区已经有未提交修改，Codex 不能假装没看见。它应该判断这些修改是否和当前任务相关。无关，就避开。相关，就在修改前说明打算怎么处理。用户已有改动不是噪声，而是现场事实。Codex 的角色是协作者，不是接管者。

第三条缰绳，是让改动保持最小范围。

用户要改一篇博客，就不要顺手改站点样式。用户要修一个脚本，就不要顺手升级依赖。用户要补一个图，就不要顺手重排整个文档。很多 agent 的问题不是做得太少，而是做得太多。它把“也许可以优化”当成“现在应该修改”。最后 diff 变大，人类审不动，风险藏在里面。

最小改动不是胆小。最小改动是可审计。一个小 diff 能看懂，能测试，能撤回。一个大 diff 只能靠摘要，而摘要最容易漏掉关键副作用。Codex 想提高效率，必须先让自己的改动变得容易检查。

第四条缰绳，是依赖和环境不能乱动。

Codex 常常会遇到缺包、缺命令、缺运行环境。它很容易为了完成任务去安装东西。问题不在安装本身，而在安装位置。项目环境里安装依赖，影响的是项目。全局安装依赖，影响的是整台机器。使用 `sudo`，影响的是权限边界。为了跑通一个局部任务，把全局环境改乱，这是很差的交换。

所以应该优先使用 Codex 当前环境或项目已有的 Python 和 Node。项目有虚拟环境，就用虚拟环境。项目有 `package-lock.json`，就尊重它。真的需要安装依赖，也要先说明原因、安装位置和影响范围，并优先安装到项目环境里。没有用户明确同意，就不要全局安装，不要使用 `sudo`。

第五条缰绳，是命令要有目的。

Codex 运行命令前，至少要知道自己在哪个目录。对可能写文件、联网、下载依赖或长时间运行的命令，还要先说明目的和预期输出。比如它要跑构建，就说这是为了验证文章能不能被站点编译。它要跑测试，就说测试覆盖哪个改动。它要联网，就说是为了查官方文档还是下载依赖。这样用户知道它在做什么，也知道失败意味着什么。

这条规则看起来细，其实很关键。agent 最危险的地方不是它运行了一个命令，而是你不知道它为什么运行这个命令。目的说清楚，命令就有了边界。目的说不清楚，它就可能把探索、修复、安装、发布混在一起。

第六条缰绳，是敏感信息不能成为上下文。

`.env`、API key、token、密码、私有数据，这些东西不要读取、不要打印、不要提交。需要检查密钥时，只检查变量是否存在。需要排查认证问题时，先看变量名、加载路径、错误码和权限范围，不要把密钥本身送进对话。很多泄漏不是一次故意公开，而是调试时顺手打印，日志里留一份，提交里又带一份。Codex 必须在这里停下来。

第七条缰绳，是视觉和项目规则要先读源文件。

涉及图表、slides、poster、UI 或文档版式时，不要直接生成。先读 `DESIGN.md`。没有设计源，就看现有样式和项目约定。视觉产物不是孤立的漂亮图片，它要属于这个项目。涉及项目规则、构建、测试、依赖或目录约定时，先读 `AGENTS.md`。不要让 Codex 从一般经验里猜项目规则。项目越重要，猜测越不应该成为默认路径。

第八条缰绳，是完成以后必须交代清楚。

Codex 不能只说“完成了”。它应该告诉你实际改了哪些文件，实际运行了哪些命令，验证结果是什么，还有什么剩余风险。没有运行的测试不能写成已通过。构建失败不能被包装成“应该没问题”。没有验证的事实不能写成事实。这个汇报不是客套，而是你判断能不能提交、推送、发布的依据。

如果任务无法安全完成，就直接说明阻塞原因。比如必须读取敏感文件才能继续，工作区已有冲突改动，依赖安装会污染全局环境，远程发布需要权限但当前没有认证。这时正确动作是停下来，而不是猜测性绕过限制。绕过限制看起来聪明，其实是在剪断缰绳。

把这些规则写进 `AGENTS.md` 以后，Codex 不会因此变得保守无用。相反，它会变得更能用。因为你不再需要每次盯着它的手。它知道哪些动作可以直接做，哪些动作要先说明，哪些动作必须拒绝。真正的效率不是让 agent 无限制奔跑，而是让它在清楚边界内持续前进。

很多人第一次使用 Codex，会被它的速度吸引。它能很快读懂目录，能很快写出 patch，能很快跑构建，能很快把一个模糊想法变成文件。这个体验会让人想继续加权限。既然它能改一个文件，那能不能让它改一批文件。既然它能跑一次命令，那能不能让它自己判断需要跑什么。既然它能生成文章，那能不能让它直接提交并发布。这个方向很自然，但也正是风险开始变大的地方。

工具越强，越需要边界。锤子只会敲，所以你不太担心锤子自己重构项目。Codex 不一样。它会读上下文，会推断目标，会选择路径，会在失败后尝试另一个路径。这个能力非常有用，但它也意味着错误不会停在第一步。如果缺依赖，它可能尝试安装。如果测试失败，它可能尝试修改测试。如果文件冲突，它可能尝试覆盖。如果目录混乱，它可能尝试清理。一个动作后面会接另一个动作，直到它以为任务完成。

所以缰绳的作用不是限制第一步，而是限制连续动作。你要告诉 Codex，哪些地方遇到阻力时可以继续，哪些地方遇到阻力时必须停。缺少普通开发依赖，可以说明原因并请求安装。发现 `.env`，必须停。看到未提交改动，必须先判断和说明。需要删除文件，必须改成归档。准备推送远端，必须确认范围。这样的规则把“继续尝试”的冲动变成一组明确分叉。

这也是为什么我不建议只在 prompt 里临时提醒。临时提醒像出门前对司机说一句“开慢点”。有用，但太笼统。真正有用的是交通规则、刹车、仪表盘和路标。全局 `AGENTS.md` 就是交通规则。项目 `AGENTS.md` 就是当前道路的路标。git 状态、构建结果、测试输出，就是仪表盘。归档目录和最终汇报，就是回看路线的记录。

如果你只写“请谨慎”，Codex 不知道谨慎具体意味着什么。它不知道清理时能不能删文件，不知道失败时能不能装依赖，不知道看到密钥时能不能打印，不知道用户已有改动是不是可以覆盖。它可能会努力表现得谨慎，但谨慎会变成一种语气，而不是一种行为。安全不能停留在语气上。安全必须落到动作上。

好的 `AGENTS.md` 应该像一份驾驶协议，而不是一篇价值观宣言。它不需要长篇大论地说“安全很重要”。它只需要写清楚，遇到某种情况，做什么，不做什么。如果需要移除文件，就移动到 `tmp/archive_YYYYMMDD_HHMMSS/`。如果需要安装依赖，就先说明位置和影响范围。如果需要检查密钥，就只检查变量是否存在。如果无法安全完成，就停下来报告阻塞原因。这些句子很普通，但普通才好执行。

还有一个常见误区，是把 Codex 当成单次任务工具。你今天让它写文章，明天让它修代码，后天让它整理实验结果。每次看起来都是独立任务，但它们共享同一台机器、同一套环境、同一个文件系统、同一个 Git 历史。一次全局安装会影响后面的任务。一次误删会影响后面的项目。一次泄漏会进入日志和提交。agent 的动作会留下状态，所以规则也必须跨任务稳定存在。

这就是全局规则的价值。它不是为了替代你的判断，而是为了让你的判断默认生效。你不需要每次都说“不要删除”。你不需要每次都说“不要打印 token”。你不需要每次都说“不要覆盖我已有改动”。这些底线应该一直在场。只有这样，你才能把注意力放回真正的任务，而不是反复提醒工具不要伤到你。

项目规则则解决另一个问题：不同项目的正确动作不一样。写博客时，发布意味着 `draft: false`、构建通过、推送触发 Pages。写论文时，发布可能意味着编译 PDF、检查引用、不要改实验数据。写小说时，发布可能意味着同步剧情状态、不要覆盖已存在章节。写前端时，视觉改动可能必须先看 `DESIGN.md`。如果没有项目规则，Codex 只能用通用经验猜。而通用经验在具体项目里经常是错的。

这不是 Codex 独有的问题。任何协作者进入一个项目，都需要知道边界。新人要读 README，要看贡献指南，要知道测试命令，要知道哪些目录不能动。Codex 只是把这个问题放大了。因为它动作更快，也更愿意在你没有说清楚时继续推进。对人类协作者，含糊规则会带来沟通成本。对 agent，含糊规则会直接变成文件系统里的副作用。

所以不要等事故发生后再补规则。真正有用的规则，应该在第一次给 Codex 写权限之前就存在。你不需要一次写完所有内容。先写底线。用几次以后，再把真实遇到的问题补进去。某次它差点改了无关格式，就加上“不改无关格式”。某次它想全局安装，就加上“不随意安装全局依赖”。某次它没有报告测试没跑，就加上“没有运行的测试不能写成已通过”。规则不是一次性作品，而是工作流的记忆。

这里也要避免另一种极端：把 `AGENTS.md` 写成百科全书。太长的规则会让关键边界被淹没。全局文件尤其应该短而硬。它管底线，项目文件管细节，临时 prompt 管这次任务的目标。三层分开，Codex 才知道什么是永远不能破的规则，什么是这个项目的约定，什么是这一次要做的事情。

当这三层分清楚以后，Codex 的行为会稳定很多。它不会因为你一句“清理一下”就直接删除。它不会因为测试缺包就擅自污染全局环境。它不会因为文章写完了就默认可以发布。它也不会把完成任务后的汇报写成一句“已完成”。它会更像一个知道边界的执行者，而不是一匹看见空地就往前冲的野马。

可以把一个真实工作日想象得更具体一点。早上你让 Codex 改一篇博客，中午让它整理一个实验脚本，晚上让它修一个 GitHub Pages 构建问题。每个任务本身都不大，但它们都在同一台机器上发生。如果第一件事里它顺手改了站点样式，第二件事里它全局装了一个 Python 包，第三件事里它为了修构建又改了依赖锁文件，那么一天结束时，你得到的不是三次提效，而是一团说不清的状态。

这就是 agent 工作和普通问答最大的不同。问答的错误留在文本里，执行的错误留在环境里。文本错了，删掉重写。环境错了，要追踪路径，要看 diff，要找缓存，要查版本，要恢复文件。你本来是想省时间，结果时间花在了修复副作用上。很多人觉得 agent 不可靠，其实是在为没有边界的执行付账。

好的规则会把这种状态污染挡在前面。它让 Codex 在改动前先说现场，在动依赖前先说范围，在遇到删除前改成归档，在遇到发布前先验证。它不会让错误消失，但会让错误小得多，也更早暴露。一个小错误可以被纠正，一个不可逆错误只能被追悔。

我也不认为这些规则会让 Codex 变笨。恰恰相反，清楚的边界会让它少浪费推理在猜测上。它不需要猜能不能删文件，因为规则说不能。它不需要猜能不能读 `.env`，因为规则说不能。它不需要猜能不能装全局依赖，因为规则说先不要。它把注意力留给真正需要判断的地方：任务目标是什么，现有代码怎么组织，最小改动在哪里，验证应该怎么跑。

对用户来说，这也会改变使用 Codex 的方式。你不再需要每次都紧张地盯着它要跑什么命令。你可以先建立底线，再把任务交出去。它报告当前目录和 git 状态，你就知道它站在正确地方。它说明会改哪些文件，你就知道影响面。它最后列出命令和验证结果，你就知道这次工作有没有闭环。信任不是因为它不会错，而是因为错了也能看见、能停住、能恢复。

全局 `AGENTS.md` 还有一个更隐蔽的好处：它会迫使你先想清楚自己的工作方式。你到底允许不允许 agent 删除文件。你到底愿不愿意它全局安装依赖。你到底希望它在发现未提交改动时怎么处理。很多人没有写规则，不是因为规则没用，而是因为自己也没想清楚边界。写下来以后，Codex 变得更可控，人自己的流程也变得更清楚。

这件事不需要复杂工具链。先从一个短文件开始就够了。把你真正不能接受的事故写进去，把你希望 Codex 每次都执行的检查写进去，把你希望它完成后必须汇报的内容写进去。用一段时间后，哪里不够，再补哪里。AGENTS.md 最好是活的，但不要每天乱变。它应该像驾驶习惯，稳定、清楚、可重复。

也不要把这件事想得太玄。给 Codex 套缰绳，不是在研究什么宏大的 AI 治理框架，而是在保护自己的工作目录。你不需要先设计完美制度。你只需要先把最容易出事的地方拦住。文件不要删，密钥不要读，依赖不要乱装，用户改动不要覆盖，发布不要假装已经验证。只要这几条在，很多事故就会从不可恢复变成可恢复，从看不见变成看得见。

等这些底线稳定以后，你可以再让 Codex 做更多事。让它写文章，让它跑实验，让它整理项目，让它修 CI，让它生成 slides。它能跑得更远，但你仍然握着方向。缰绳不是不信任马，而是承认马有力量。越有力量，越需要方向。

如果你现在还没有全局 `AGENTS.md`，不要等下一次事故。今天就写。先写十几行也可以，先覆盖最危险的动作也可以。真正重要的是让规则先进入系统，而不是继续停留在脑子里，停留在下一次口头提醒里，最后又被遗忘，也被持续反复误解。

这就是我说的提效。不是让 Codex 做更多动作，而是让它做更少错误动作。不是让它更自由，而是让它更可控。一个可控的 agent，才能被放心地交给更大的任务。一个不可控的 agent，只适合做低风险玩具。你不敢让它碰重要文件，不敢让它碰真实项目，不敢让它碰发布流程，那它再快也只是演示。

所以问题不是“Codex 会不会误操作”。任何能执行的系统都会误操作。真正的问题是，误操作发生前有没有边界，发生时有没有停机条件，发生后有没有恢复路径。`AGENTS.md` 不是万能保险，但它是最低成本、最容易落地的控制面。先把缰绳套上，再让 Codex 跑。

- 改动应保持最小范围. 只修改完成当前任务必要的文件, 不做顺手重构, 不改无关格式, 不移动无关目录.

- 涉及图表、slides、poster、UI 或文档版式时, 先读取 DESIGN.md, 再生成或修改视觉产物.

- 涉及项目规则、构建、测试、依赖或目录约定时, 先读取 AGENTS.md, 并遵守其中的命令和边界.
