社交媒体上那些 agent 误操作的故事，表面看起来像模型失控。文件被删了，目录被清空了，依赖被全局装乱了，系统路径被改坏了。每个故事都有戏剧性，但真正的原因并不戏剧。不是模型突然变坏，也不是智能本身有魔法。问题更朴素：用户把一个能读文件、写文件、跑命令、联网、安装依赖的执行者放进系统，却没有先规定它什么时候必须停下来。

Codex 的危险不在于它会回答。危险在于它会执行。一个只会说话的模型说错了，你还能不采纳。一个能操作工作区的 agent 误解了任务，它可能已经改了文件、覆盖了手稿、提交了无关 diff，甚至把你原本不想公开的内容带进输出。把这个问题说成“AI 不可靠”太粗。真正的问题是，执行能力没有被放进制度里。

所以在谈提效之前，要先谈缰绳。不是为了让 Codex 变慢，而是为了让它能安全地变快。没有缰绳的野马不是自由，它只是不可控。任它跑，未必能抵达目的地，却一定会把风险转嫁给驾驶者。Codex 也一样。你不能一边给它 shell、git、文件系统和网络，一边只靠一句“请小心”来管理副作用。

## AGENTS.md 不是装饰

很多人把 agent 指令当成口味设置。语气要简洁，回答要中文，写代码要优雅。这些都可以写，但它们不是最重要的。真正重要的是工作边界。哪些文件不能动，哪些命令不能跑，哪些依赖不能装，遇到未提交改动时怎样处理，看到密钥时怎样退出，完成任务后怎样报告风险。它们不是风格偏好，而是安全协议。

OpenAI 的 Codex 文档把 `AGENTS.md` 定义为项目自定义指令的入口，并说明 Codex 会在开始工作前读取这些文件，再按全局、项目和更深目录的顺序合并指导。全局文件通常放在 `~/.codex/AGENTS.md`，项目文件放在仓库根目录或子目录。离当前工作目录更近的文件，会在合并后的指令里更靠后，因此可以覆盖更宽泛的规则。这个机制的意义很直接：不要把每次对话都当成第一次谈判。把稳定的工作契约放进文件，让每个会话从同一组边界开始。

但这里有一个容易犯的错。全局 `AGENTS.md` 不是万能保险。它能提供默认约束，却不能替代项目规则。一个小说项目、一个科研代码库、一个静态网站、一个部署仓库，风险面完全不同。全局文件应该写跨项目原则，项目文件应该写本项目的目录约定、构建命令、测试命令、发布边界和视觉规范。再深一层的目录如果有特殊规则，就把更具体的规则放在那里。缰绳不是一根绳子，而是一套分层控制。

## 全局规则管底线

全局 `AGENTS.md` 只应该管那些不随项目变化的底线。不要删除文件。不要泄漏密钥。不要覆盖用户改动。不要全局安装依赖。运行有副作用的命令前说明目的。完成后报告文件、命令、验证和风险。这些规则不需要知道你在写小说、跑实验还是维护网站。它们管的是 agent 进入任何工作区时都必须遵守的生存协议。

项目 `AGENTS.md` 管的是局部事实。这个项目用 `npm run build` 还是 `make test`。内容写在 `src/content/blog` 还是 `docs/posts`。图像产物要不要先读 `DESIGN.md`。论文图表输出到 `figures/` 还是 `assets/`。实验结果能不能覆盖旧 CSV。发布是不是通过 `main` 分支触发 GitHub Pages。项目规则越具体，Codex 越不需要猜。它少猜一次，就少一次越界机会。

把全局规则写成项目细节，会让它变脆。今天你在 Astro 站点里写博客，明天你在 Python 包里跑实验，全局文件如果塞满前一个项目的命令，就会污染下一个项目。反过来，把项目细节留在每次 prompt 里，也不可靠。人会忘，agent 会压缩上下文，对话会跨越多个任务。稳定规则应该落到稳定文件里。临时口头授权只适合一次性例外，不适合长期边界。

这一点也是很多事故复盘里被忽略的部分。用户常以为自己“已经说过要小心”，但 agent 实际看到的是当前上下文，不是人的长期记忆。把规则写进文件，是把人的期望变成机器可重复读取的输入。它不能保证每次都完美遵守，却能让偏离变成可指出、可纠正、可回归测试的问题。

## 第一条规则是不删除

最危险的命令通常不是复杂命令，而是太简单的命令。`rm -rf` 看起来干净，结果也干净。它不问你那是不是误判，不保留上下文，不给恢复窗口。对人来说，删除是一个明确动作。对 agent 来说，删除常常只是完成“清理一下”的一种路径。你说清理，它可能理解成移除。你说重构，它可能理解成替换。你说整理，它可能理解成删掉旧版。

所以第一条全局规则应该非常硬：永远不要直接删除文件。需要移走的东西，放进当前项目的 `tmp` 目录，并打上时间戳，例如 `tmp/archive_YYYYMMDD_HHMMSS/`。这条规则的价值不只是避免损失。它还把不可逆动作改成可审计动作。文件还在，只是离开主路径。你可以 diff，可以恢复，可以确认它确实不再需要。任务完成后，真正要不要永久删除，由人决定。

这条规则听起来保守，但它符合工程常识。数据库迁移不会先丢表再祈祷。发布系统不会在没有回滚包的情况下覆盖线上版本。文件系统里的草稿、实验结果、图像源文件、旧脚本也一样。agent 没有资格把一个模糊意图直接翻译成不可逆删除。它最多有资格隔离、归档、标记和报告。

## tmp 不是垃圾桶

把文件移动到 `tmp/archive_YYYYMMDD_HHMMSS/`，不是把垃圾换个地方堆。这个目录应该是恢复缓冲区，也是审计缓冲区。每一次归档都应该让人看得出发生了什么。时间戳说明动作发生的时间。目录名说明归档属于哪次任务。最终汇报说明哪些文件被移动、为什么移动、是否还被引用。这样，用户看到的不是“东西没了”，而是“东西被隔离到这里，原因是这个，恢复路径是这个”。

这比单纯禁止删除更实用。实际工作里确实需要清理。旧构建产物会污染搜索结果，过期草稿会干扰发布，重复图片会让目录不可读，实验中间文件会拖慢同步。完全不移动文件，会让项目越来越重。问题不在清理，而在清理动作必须可逆。可逆性让 agent 可以参与整理，又不获得永久销毁权。

`tmp` 规则还会改变 Codex 的推理方式。没有这条规则时，它会把“清理”理解成找到不需要的文件并删除。有了这条规则，它必须先判断哪些文件是候选，再选择隔离路径，再报告隔离结果。这个过程天然增加了检查点。人可以在检查点纠正它。比如某个旧脚本其实是复现实验的入口，某个临时 Markdown 其实是尚未发布的笔记。错误仍可能发生，但错误被限制在可恢复范围内。

最好的 `tmp` 目录还可以配一个轻量说明文件。不是每次都要写长日志，但至少可以记录任务名、时间、移动原因和恢复提示。这样三天后再看，归档目录不是一堆失忆文件，而是一个可以理解的现场快照。对 agent 来说，这也是训练自己尊重状态的方式。它不是在替用户扔东西，而是在把不确定对象放到待确认区。

## 第二条规则是先看现场

Codex 进入一个目录时，不能假设它是干净的。它必须先确认自己在哪里，当前 git 状态是什么，哪些文件已经被修改，哪些改动不属于这次任务。很多误操作不是因为模型不会写代码，而是因为它在错误现场写了正确代码。它覆盖了用户刚刚改过的文件，格式化了无关模块，或者把另一个任务留下的中间产物当成垃圾清掉。

所以修改文件前，先检查当前目录和 git 状态。这个动作不是仪式。它告诉 Codex 三件事。第一，当前任务的工作根在哪里。第二，工作区里哪些东西已经脏了。第三，接下来哪些文件可能被读取，哪些文件可能被修改。如果已有未提交改动，Codex 应该先判断它们是否与当前任务相关。无关就避开。相关就说明处理方式，再进入修改。

这也是为什么“不要覆盖用户已有改动”必须写成硬规则。用户的未提交改动不是噪声，而是现场事实。agent 的默认姿态应该是协作，而不是接管。它可以在同一个文件里追加自己的改动，但前提是先读懂已有改动的意图。它不能为了让自己的 patch 更容易应用，就把别人的编辑还原到旧状态。那不是整理工作区，那是破坏现场。

## 最小改动是安全策略

很多 agent 事故不是一次大爆炸，而是一串小越界。用户要改一处文案，agent 顺手重排整个文件。用户要修一个测试，agent 顺手升级依赖。用户要新增一篇博客，agent 顺手改样式、移动目录、格式化历史文章。每一步看起来都“也许有帮助”，合起来就变成不可审查的 diff。

因此最小改动不是审美偏好，而是安全策略。完成当前任务需要改哪些文件，就只改这些文件。能不碰格式就不碰格式。能不移动目录就不移动目录。能不改共享抽象就不改共享抽象。真正需要扩大范围时，先说明为什么窄改不够，再说明扩大后的影响面。这个顺序很重要，因为它迫使 agent 把“我想顺手做”翻译成“这是完成目标的必要条件”。

最小改动还保护验证。一个小 diff 可以被人看懂，可以被测试覆盖，可以被撤销。一个大 diff 会制造审查疲劳。人类最后只能看摘要，而摘要正是最容易遗漏事故的地方。Codex 要成为可靠工具，就必须让自己的工作结果保持可读、可审、可回滚。

## 依赖安装必须有边界

另一个常见风险是依赖。agent 为了让脚本跑起来，很容易安装缺失包。问题不在安装本身，而在安装位置。全局安装改变的是整台机器。`sudo` 改变的是权限边界。把依赖装到系统 Python、全局 npm 或用户级工具链里，可能解决当前任务，却污染未来所有任务。短期看它很高效，长期看它是环境债务。

更好的规则是优先使用 Codex 当前环境或项目已有的 Python 和 Node。项目有虚拟环境，就用虚拟环境。项目有 `package-lock.json`，就尊重锁文件。项目有本地脚本，就先读脚本。只有在确实缺少必要依赖时，才向用户说明安装原因、安装位置和影响范围。默认安装到项目环境，不使用 `sudo`，不做全局安装，除非用户明确同意。

这条规则背后的原则是作用域控制。一个任务需要的能力，应该尽量局部化。局部依赖坏了，坏的是项目环境。全局依赖坏了，坏的是机器习惯。agent 不是系统管理员，不应该把完成一个局部任务的便利扩展成全局状态变化。

## 命令也需要说明

人运行命令时，通常知道自己为什么运行。agent 运行命令时，用户只能从结果倒推意图。这个不透明性会制造风险。尤其是写文件、联网、下载依赖、启动长任务、触发构建、提交、推送和部署。这些命令不是普通查询，它们会改变状态，消耗时间，或把本地信息带到外部。

所以运行命令前必须确认当前目录。对可能写文件、联网、下载依赖或长时间运行的命令，先说明目的和预期输出。这个说明不需要啰嗦，但必须具体。比如构建命令是为了验证新博客能被 Astro 编译，预期输出是 build 成功或指出 frontmatter、Markdown、类型检查错误。质量门是为了扫描文章里的模板腔和结构问题，预期输出是通过或给出可修订警告。这样用户知道 agent 在做什么，也知道失败意味着什么。

命令边界还包括不要把测试结果写成想象。没有运行的测试不能说通过。构建失败不能被“应该没问题”掩盖。网络请求失败不能假装来源已经验证。可靠的 agent 不靠自信，而靠记录。

## 敏感信息不是上下文

`.env`、API key、token、密码、私有数据，这些东西不应该成为模型上下文。不是因为模型一定会泄漏，而是因为没有必要。需要检查密钥时，只检查变量是否存在。需要验证配置时，输出键名和存在状态，不打印值。需要排查认证失败时，先看错误码、权限范围、加载路径和环境变量名，而不是把秘密本身交给模型。

这条规则要写进全局文件，因为人在紧急调试时很容易忘。用户会说“看看这个 env 对不对”。agent 如果没有硬边界，就可能把整个文件读出来。更糟的是，它可能把密钥带进日志、审计、最终回复或 commit diff。敏感信息一旦进入多层工具链，就很难证明它没有扩散。

安全做法是数据最小化。只读完成任务所需的最小信息，只输出用户需要判断的最小事实。存在性、路径、权限、格式、过期状态，通常已经足够。真实秘密不该进入对话，也不该进入提交。

## 视觉产物要服从设计源

涉及图表、slides、poster、UI 或文档版式时，Codex 不应该直接开始画。视觉产物不是孤立输出，它属于一个设计系统。颜色、字体、布局密度、图像风格、品牌约束、阅读顺序，都可能已经写在 `DESIGN.md` 里。跳过设计源，agent 生成的东西也许漂亮，却不一定属于这个项目。

所以视觉任务必须先读取 `DESIGN.md`。没有这个文件时，也要先寻找相邻的设计说明、现有页面、样张或组件约定。这个动作不是拖慢创作，而是防止“漂亮的错误”。在工程工作里，视觉一致性也是一种约束。一个不符合系统的 poster 会增加修改成本。一个不符合组件规范的 UI 会污染前端结构。一个不符合论文图风格的图表会削弱整体叙事。

同理，涉及项目规则、构建、测试、依赖或目录约定时，先读取项目的 `AGENTS.md`。全局规则告诉 Codex 怎样做人，项目规则告诉它怎样在这个项目里做事。两者缺一不可。

## 交付必须包含证据

一个 agent 完成任务后，只说“完成了”是不够的。用户真正需要的是可验证交付。改了哪些文件，运行了哪些命令，验证结果是什么，哪些风险还剩下。没有这些信息，人类就无法判断是否可以继续发布、提交或上线。

这也是区分可靠 agent 和热心脚本的地方。可靠 agent 会把自己的行动留下轨迹。它不会把没运行的测试写成已通过，不会把未验证的来源写成事实，不会把构建警告藏在摘要里。它会说明实际改动文件，实际运行过的命令，验证通过或失败的具体结果，以及剩余风险。剩余风险不等于失败。它只是承认系统边界还在。

如果任务无法安全完成，也要直接说明阻塞原因。比如工作区已有冲突改动，敏感文件必须读取才能继续，缺少依赖但用户不同意安装，发布需要远程权限但认证失败。正确做法是停下来，而不是猜测性绕过限制。绕过限制看似聪明，实则破坏了缰绳本身。

## 发布门要比生成门更硬

生成一篇文章、改一段代码、做一张图，都只是候选产物。提交、推送、部署、发邮件、改数据库、发布到公网，才是把候选产物变成外部事实。很多 agent 工作流的问题，是把这两类动作混在一起。用户说“帮我写”，agent 以为可以顺手发布。用户说“修一下”，agent 以为可以顺手提交。用户说“跑通”，agent 以为可以顺手安装和改配置。这里真正缺的不是能力，而是门。

发布门应该比生成门更硬。生成阶段可以探索，可以试错，可以留下草稿。发布阶段必须要求更明确的证据。博客发布要有 frontmatter、正文、私有审计、质量门、构建结果和可解释 diff。代码提交要有测试或说明为什么无法测试。推送远端要确认当前分支、远端地址和提交范围。部署要确认它会改变公网状态。越靠近外部世界，越不能靠“应该没问题”。

这条规则会让 agent 的节奏更像成熟工程流程。先生成，再验证，再交付，再发布。每一步都可以停。停不是失败，而是控制。没有停机点的自动化才危险，因为它把所有中间判断都折叠成一个最终动作。等你发现错了，文件已经删了，提交已经推了，页面已经发布了，包已经传了。

更细一点说，发布门还要区分本地状态和远端状态。本地新增文件可以撤销。本地 commit 可以重写。已经 push 到公开仓库，影响面就变了。已经触发 GitHub Pages 或包发布，影响面更大。已经把密钥、私有数据或错误声明推到公网，补救成本远高于本地修正。Codex 必须知道这些状态差异，用户也要把这些差异写进规则。

## 权限不是信任的替代品

给 agent 权限很容易。真正困难的是给权限配套约束。文件系统权限让它能读写，并不等于它应该读写。网络权限让它能访问外部，并不等于它应该把本地信息发出去。shell 权限让它能运行命令，并不等于它应该把每个失败都用安装依赖解决。权限只是能力开关，不是任务判断。

这也是为什么“我相信 Codex”不是一个足够好的工作策略。你相信的是它在明确边界内能把事情做好，不是相信它在边界缺失时自动替你设计制度。人对工具的信任来自可预测性。可预测性来自规则、反馈和可恢复性。没有这些东西，信任会退化成侥幸。

一个好的全局 `AGENTS.md` 会把信任拆成几个可检查条件。它让 Codex 在改文件前暴露现场，让它在安装前说明影响，让它在敏感信息前减小输入，让它在发布前提供证据。你不是因为它像人一样谨慎才信它，而是因为它被要求用一套可观察动作证明自己谨慎。

## 全局规则应该写得像协议

很多人写指令喜欢写愿望。“请谨慎操作。”“尽量不要破坏文件。”“注意安全。”这些话对人有用，对 agent 太软。协议应该写成可执行的条件和动作。遇到删除需求时，移动到 `tmp/archive_YYYYMMDD_HHMMSS/`。安装依赖前，说明原因、位置和影响范围。发现未提交改动时，判断相关性并避开无关改动。需要密钥时，只检查变量是否存在。完成后，报告文件、命令、验证和风险。

这样的句子有一个好处：它们能被审查。你可以问 Codex 是否遵守了它。你可以在最终汇报里检查它是否列出了文件和命令。你可以在 git diff 里确认它有没有动无关文件。你可以在失败时指出具体违反了哪条规则。软指令只能制造愧疚，硬协议才能形成反馈。

这并不意味着全局 `AGENTS.md` 要写得很长。恰恰相反，它应该短、硬、稳定。把跨项目底线放进去，把项目细节留给项目文件。不要把所有个人偏好、写作风格、工具说明、项目知识都塞进全局文件。Codex 文档也提醒，项目指令有大小限制；指令过长会增加成本，也会把关键规则埋在噪声里。缰绳要结实，不要缠成一团。

协议还应该写清失败动作。发现敏感信息时不是“尽量保护”，而是停止读取并改为检查变量存在性。发现工作区冲突时不是“尝试合并”，而是报告冲突文件和建议路径。发现测试无法运行时不是“略过”，而是说明缺少什么、已经验证了什么、还没验证什么。失败动作越清楚，agent 越少需要临场发挥。临场发挥是智能的优势，但在安全边界上，它不应该替代制度。

## 真正的提效来自可控

很多人使用 agent 时，第一反应是给更多权限。让它自己搜，让它自己改，让它自己装，让它自己跑。权限越多，短期演示越顺。可一旦进入真实工作，问题会反过来。你不敢让它碰重要目录，不敢让它处理含密钥的项目，不敢让它提交，不敢让它发布。没有边界的自由，最后会降低信任半径。

给 Codex 套上缰绳，不是把它降级成问答工具，而是让它有资格进入更严肃的工作流。能归档而不删除，能最小改动，能尊重未提交现场，能把依赖限制在项目里，能在命令前说明目的，能避开敏感信息，能按设计源生成视觉产物，能在完成后交付证据。这样的 agent 才能长期使用。它不需要每次都靠用户盯着手。

所以正确的问题不是“Codex 会不会误操作”。任何能执行的系统都会误操作。正确的问题是，误操作发生前有没有边界，发生时有没有停机条件，发生后有没有恢复路径。`AGENTS.md` 不是银弹，但它是最低成本、最高频、最容易被忽视的控制面。先写它，再谈提效。先把缰绳套上，再让马跑。
